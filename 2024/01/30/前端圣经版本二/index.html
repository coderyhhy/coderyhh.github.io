原型相关

  1. 是什么？

    • 所有的实例对象都具有隐式原型 （_proto_ ）
    • 所有函数都有显示原型（prototype）
    • 实例对象的隐式原型只想构造函数的显示原型
    • 所有的原型对象都具有 constructor 属性，指向对应的构造函数。

    总结：一个原型对象，两个原型属性。

  2. 为什么要用？

    • 直接目的：可以让原型对象身上的数据，共享给所有实例对象使用。
    • 最终目的：可以将本来每个实例对象身上相同的方法，全部提取到组合成一个函数，放在原型对象上，用于节省内存空间。

  3. 怎么用？

    • 将数据放在原型对象身上，实例对象如果使用该属性，自身如果没有就会顺着原型链找到该原型对象，从它身上获取该属性。
    • 读取属性就会顺着原型链向上查找，设置属性不会顺着原型链向上查找，会直接复制给原型。

  4. 在哪儿用过？

    • 在 Vue 全局事件总线

每次执行依次匿名函数声明，就会创建一个全新的匿名函数。

new 做的事情

  • 自动声明形参变量，并将传入的实参数据赋值给对应的形参。
  • 自动声明 this 变量，并传入一个全新的实例对象。
  • 将当前构造函数的显示原型属性值，赋值给实例对象的隐式原型属性。
  • 默认 return 当前实例对象

在 JS中, 用 C语言实现的东西:Object Function Function.prototype 这三个

在 JS 中, 强行修改了四条线

  1. Function._propop_ === Function.prototype
    • 身份证明
      • 如果没有这条线,那么 Function就不在是函数了
    • 使用方法
      • 所有函数共享的方法都会放在 Function.ptototype 上, 如果没有这条线, 那么 Function 将无法使用 call bind apply 等等方法
  2. Object._proto_ === Function.prototype
  • 同上
  1. Function._proto_ === Object.ptoyotype
    • 如果没有这条线, 那么原型链将会出现两个终点, 那么函数就不在是对象了
  2. Object._proto_ === null
    • 如果没有这条线, 那么原型链将会进入死循环, 无论是那个实例对象查找属性, 都会在 Object 卡住

作用声明域

在编程界, 作用域一共分为两种

  1. 词法作用域 (静态作用域)
    • 函数的作用域声明位置有关, 与调用位置无关, JS 就是静态作用域
  2. 动态作用域
    • 函数的作用域与声明位置无关, 与调用位置有关

在 JS 中, 只有词法作用域, 没有动态作用域

作用域就是用来约束变量的作用范围, 作用域其实就是变量的查找规则

预解析规则(变量提升)

  1. 变量提升
    • 变量提升是局部提升, 它会将标识的声明提升到最顶端, 标识的赋值会留在原地
  2. 函数提升
    • 函数提升是整体提升, 声明和赋值都会提升到最顶端

注意: 区分清楚标识声明和标识赋值

例如 var a = 1

  • 标识声明 var a
  • 标识赋值 a = 1

函数表达式提升的是变量提升, 只会将标识声明提前, 赋值将会留在原地

预解析的优先级: 函数优先, 函数提升优先于变量提升

预解析的范围:

  • 会将标识声明提升到当前作用域的最顶端, 不会穿透作用域

注意: 变量查找都是从内向外查找

作用域分类

  1. 全局作用域
    • 它是 JS 最大的顶级作用域
    • 所有人都可以获取到他身上的数据
    • 扩展: 全局作用域其实也是函数作用域, 因为 JS 引擎会将主线程代码放在一个匿名函数中调用执行
    • 但是它是一个最顶级的函数, 所以所有人都可以用
  2. 局部作用域 (函数作用域)
    • 与函数有关, 每个函数都有自己独立的一片作用域
    • 内部的数据不能被外侧访问
  3. 块级作用域
    • 只要使用 const let 就会出现 ES6 新增的块级作用域
    • 注意点: let 和 const 需要根据外层的大括号进行配合, 生成块级作用域
    • 在同一个大括号中, 多次使用 let const 都只会生成一个块级作用域
  4. 形参默认值( es6新增语法)
    • 只要函数使用了形参默认值, 那么函数内部的代码就会在新生成的块级作用域中执行
      • 注意: 函数内部在块级作用域中, 函数形参还在函数作用域中
      • 函数被调用, 如果函数内部和函数形参的名字相同, 那么他们的初始值也会相同, 函数形参会将自己的值传递一份给函数内部同名变量
    • 作用域是一套规则, 一套定义(并不是真正意义存在的)
      • 变量对象就是作用域实现的结果

闭包

  1. 闭包的产生条件

    • 闭包是一种引用关系
      • 必须具有内外函数嵌套
      • 内部函数必须使用外部函数的变量
    • 闭包分为两种
      • 有效闭包: 外部函数被调用的时候, 会将内部函数返回出去, 那么这个闭包就可以持续保留, 称为有效闭包
      • 无效闭包: 外部函数被调用的时候, 没有将内部函数返回出去, 那么这个闭包就会快速销毁, 称为无效闭包

  2. 闭包的创建时机

    • 在外部函数被调用的时候, 就会产生闭包
    • 一个函数一次调用只会创建一个闭包对象
      • 因为闭包对象与执行上下文有关

  3. 闭包的优点

    • 延长局部变量的生命周期
      • 外部函数的变量会存储于外部函数的执行上下文中, 但是外部函数执行结束的时候, 对应的执行上下文会销毁, 会导致存在内部的变量也被销毁
      • 但是内部函数还需要使用外部变量, 所以 JS 使用闭包将该变量缓存起来
    • 可以实现 jS模块化
      • 通过将数据存放在函数作用域中, 来保证该数据的安全性, 防止外侧其他开发者修改数据, 并且会向外提供一些修改该数据的方法

  4. 闭包的缺点

    • 由于闭包会缓存本该死去的变量, 会导致无法正常释放, 会被继续占用
    • 内存泄漏
      • 在平常实现功能的时候, 可能付出了更多内存空间才能实现某个功能
        • 简单的说就是内有管理好内存, 浪费了内存
    • 内存溢出
      • 平常浪费的内存太多了, 超过了计算机内存上限, 导致计算机宕机, 无法使用
    • 内存泄漏我们可以接受, 内存溢出肯定不能接受

  5. 如何解决闭包的副作用

    • 尽量少使用闭包

    • 及时释放不用的闭包

      • 为什么闭包不会被释放

        • 由于内部函数还需要使用闭包中的变量, 所以闭包不能被释放
        • 内部函数因为被外部变量引用着, 所以内部函数不会被释放

        注意:

        • 如果内部函数具有多个变量引用它, 需要将全部都释放掉才可以
        • 多个内部函数都是用到外部函数的变量 , 那么他们会共享一个闭包对象, 这个闭包对象中, 会存放所有缓存变量
        • 如果多个内部函数都在引用闭包对象, 记得要所有的内部函数都释放掉

    • 方法

      • 将存储内部函数的变量值更新, 将内部函数对的引用进行释放, 内部函数被释放, 就会导致闭包无人使用, 那么闭包也会被释放 比如 (fn = null)

  6. 闭包是什么

    • 闭包是一个对象
      • 闭包内部会存储需要使用到变量和它的变量值
    • 官方定义
      • 闭包是外部函数作用域和内部函数的词法环境组成
    • 简单点说
      • 闭包是外部函数的变量对象和内部函数的代码生成的对象

  7. 闭包的使用场景

    • 防抖节流
    • 删除后台管理, 品牌商品功能

垃圾回收机制

  • 如果一个对象没有人使用 / 引用它, 垃圾回收机制就会将他占用的内存释放

执行上下文

  • 执行上下文中会存放当前函数中 this 指向, return 结果, 以及变量对象
    • 变量对象是用来收集当前函数中, 声明的所有变量以及变量值得对象
  • 创建时间
    • 在函数调用的时候, 就会创建一个全新的执行上下文
      • 这个执行上下文推入执行栈中, 这个过程俗称 (入栈)
      • 栈是数组实现的, 先进后出, 后进先出
  • 销毁时机
    • 在函数调用结束的时候, 就会销毁对应的执行上下文
      • 这个过程俗称 (出栈)

this 指向

this 能判断出函数中的 this 是谁

能控制函数 this 是需要的特定对象

this

  • 它是 JS 引擎的关键字, 也就是禁止开发者声明, this 它的本质就是一个变量
  • this 中的值, 与函数调用方式有关, 与声明位置无关, 调用位置都无关
  • this 很像形参, 像是浏览器专用的形参
    • 形参: 函数自动声明 形参的值是开发者传入的
    • this: 函数自动声明 this 的值是浏览器传入的

常见的this 指向

  • this 内部存储的是一个对象, 这个对象代表着当前函数的调用者
  • 普通调用 window
    • 例如 fn()
    • 严格模式下, 这个 this 是undefined
    • 普通模式下, this 如果是 undefined 就会被修改为 window
  • 隐士调用 方法的调用者
    • 例如 obj.fn()
    • 无论代码找了多少层属性, 都只要看方法是在谁身上就行
  • 构造函数调用 全新的实例对象
    • 例如 new fn()
  • 显示调用 强行指定的对象 call apply bind
    • 例如 fn.call(obj) fn.apply(obj) fn.bind(obj)
  • 比较特殊的this 指向
    • 事件回调函数
      • this 是事件源, 绑定的事件元素
    • Vue
      • methods computd watch 生命周期钩子函数等中的 this 都是当前组件的实例对象
    • 小程序
      • 事件回调函数, 生命周期钩子函数中 的this 是当前页面的实例对象
    • 定时器
      • this 是window
    • 箭头函数
      • 它的this 与声明位置有关, 它会使用上级作用域的this
        • 它没有属于自己的this
    • 请问: 箭头函数能被 call apply bind 方法改变 this 指向
      • 不能, 因为箭头函数没有 this 使用这三个方法不会报错, 但是没有效果
    • 请问: 箭头函数能否被构造调用
      • 不能 会报错 因为它不是构造器
    • 请问箭头函数是否拥有自己的原型对象
      • 没有

异步

异步: 将一件事情放到未来某个时间去做, 这就是异步 (懒), 将一个函数延迟一段时间执行, 这就是异步

什么叫回调函数

  • 我创建的
  • 我没吊用
  • 它执行了

回调函数一定是异步的吗

  • 不一定是异步的, 数组的回调函数, Promise 的回调函数都是同步执行的

JS 中异步任务分为几种

  • 宏任务
  • 微任务

异步代码一定晚于主线程代码

存储多个宏任务队列称为宏任务队列

存储多个微任务队列称为微任务队列

队列

  • 数据类型: 数组
  • 特点: 先进先出, 后进后出

栈

  • 数据类型: 数组
  • 特点: 先进后出, 后进先出

执行顺序: 先执行主线程代码 然后清空微任务队列, 执行宏任务

注意: 微任务队列都是清空, 不清空不能走, 宏任务是执行, 每一次执行完宏任务都要去看一下微任务是否有东西, 有就清空

宏任务

  • 定时器
  • 事件回调
  • ajax 相关

微任务

  • Promise 的 .then 方法
  • mutationObserver( H5 新增特性)
    • 它可以监视页面上某个节点的结构变化, 只要发生变化就会触发对应的回调函数
  • 以上这些都是开启宏任务和微任务的手段

请问 JS 中 宏任务优先还是微任务优先

  • 如果将 script 当做第一个宏任务看, 就是宏任务优先
  • 如果将 script 不当做宏任务看, 就是微任务优先

浏览器多线程

JS 是单线程, 浏览器是多线程

  • 浏览器是一个多线程多进程的东西
    • 每个标签页都是全新的进程
  • 浏览器中具有
    • 负责解析 JS 代码的线程 JS线程
    • 负责页面渲染的线程 CUI 线程
  • 以上两个线程是互斥的, 同时一个时间只能一个人工作, 要么渲染页面, 要么执行代码

页面渲染流程

  • 执行主线程代码
  • 清空微任务队列
  • GUI 线程渲染页面
  • 执行下一个宏任务

更新页面流程

  • 清空微任务队列
  • GUI 渲染页面
  • 执行宏任务

请说说对挂载和渲染的了解

  • 挂载和渲染不是同一个东西
    • 挂载: 是将节点放在 HML 节点中
    • 渲染: 根据最新的 HTML 节点绘制内容到浏览器上给用户观看
  • 挂载是同步的 渲染是异步的

问: 明明有了宏任务, 为什么还要增加微任务

  • ES6 开始才正式新增微任务
  • 如果同样修改 100 万次页面, 100 万个微任务都修改页面, 会导致页面只渲染一次
    • 微任务相当于给页面做了防抖操作, 页面只会根据最后一次的结果渲染页面
  • 如果100 万个宏任务都修改页面, 会导致页面渲染 100 万次

拓展: Vue更新 DOM 也是微任务执行的

浏览器事件轮询机制

  • JS 是通过事件循环机制来实现的 JS是单线程
  • 异步要基于回调来实现
  • event loop 就是异步回调实现的原理

事件轮训机制两个重要部分

  • 在分线程执行的管理模块, 定时器 ajax dom 事件
  • 保持待执行的回调函数的事件队列 任务队列

小总结: 浏览器中具有许多分线程帮忙, 所以 JS引擎负责的事情就相对比较少.

JS 线程会将定时器, Ajax 请求等任务交个对应的管理模块进行管理

满足条件之后, 会自动往对应的任务队列中放入回调函数

浏览器具有一个宏任务队列, 一个微任务队列

Node轮训机制

事件轮询机制主要分为六个阶段 这六个阶段是六个宏任务队列

  • timers 定时器阶段 重要 它是事件轮询的起点
    • 计时和执行点到定时器回调函数
  • pending callbacks
    • 某些操作系统. 如 lcp 错误类型的回调函数
  • idle prepare
    • 准备工作
  • poll 轮询阶段 (轮询队列) 重要 专门复杂 I / O 相关的异步任务 它是事件轮询的休息区
    • 如果轮询队列不为空, 依次同步取出轮询队列中第一个回调执行, 直到轮询队列为空或者达到系统量大的限制
    • 如果轮询队列为空
      • 如果之前设置 setImmediate函数
        • 直接进入下一个check 阶段
      • 如果之前没有设置 setImmediate 函数
        • 当前 poll 阶段等待
          • 查到了轮询队列添加回调函数, 就去第一个情况执行
          • 如果定时器到点了, 也会去下一个阶段
  • check 查阶段 重要
    • 执行 setImmediate设置的回调函数
  • close calbacks 关闭阶段
    • 执行 close事件回调函数
  • node 中一个有六个宏任务 两个微任务队列
    • 宏任务: setTimeout setInterval Ajax DOM事件监听
    • 微任务 Promise.then async / await mutationobserve(H5) processnextTick(Node)
  • 宏任务队列: 用来保存 n 个宏任务队列容器
  • 微任务队列: 用来保存 n 个微任务队列容器

定时器延迟 bug

  • 定时器的第二个参数, 代表当前函数执行的延迟时间
  • 最小值为1 即便写 0 也会变成1
  • 现象是打印结果出现了 1 2 或者 2 1
  • 原因: 因为定时器的延迟时间最小为 1ms 但是主线程的代码可能没有消耗到 1ms 导致到达timers 阶段的时候, 还没有满足执行条件, 所以会先执行 setImmediate的宏任务
  • 解决: 延长主线程代码时间, 让其超过 1ms 毫秒

node 宏任务注意点:

  • node 一共有 6 个阶段, 对应 6 个宏任务队列, 所以在 Node 查看执行顺序, 需要优先关注他们的阶段, 如果是同阶段的宏任务, 在看先后顺序
  • node 事件轮询的起点是 1 号阶段(timers 阶段) 休息区 4 号阶段( I / O阶段)
    • 事件轮询不存在跳阶段的可能性, 一定从1 走完6 在从6 回到1 不断轮询

node 微任务部分

  • node 中有两个微任务

    • promise.then

      • 它与宏任务相比是 vip 级别

    • process.nextTick

      • 他与 .then 相比是 svip级别

      nextTick 会优先于 .then 而 .then 会优先于宏任务执行

注意: node 有两个微任务队列

  • 微任务队列需要清空之后, 才能切换到下一个队列的

Promise 的理解

Promise数据类型

  • 函数(它是一个构造函数)

  • 接收参数

    • 接收个数: 1个
    • 数据类型: 函数(又称为执行器函数)
      • 同步调用执行器函数
      • 接收两个参数
        • resolve 函数
          • 可以控制当前的 p1状态返回成功
        • reject 函数
          • 可以控制当前 p1状态返回错误, 将其变为失败
          • 并且会在控制台报错
        • 返回值: 全新的Promise 实例对象

  • Promise 身上具有的重要属性

    • state 状态
      • pending 等待中
      • fulfiled 成功
      • rejected 失败
    • 注意: promise对象的状态只能改变一次, 而且不可逆
    • result 结果
      • 可以通过给 resolve 或者 reject 方法传入实参, 来进行赋值控制

  • .then

    • 用处: 可以监视某个 promise 对象状态变化, 并返回对应的回调函数

    • 数据类型: 函数

    • 接收参数: 两个

      • 数据类型: 函数
      • 第一个实参是成功的回调
        • 如果监视的promise 变为成功状态, 那么就执行该回调函数
      • 第二个实参是失败的回调
        • 如果监视的promise 变为失败状态, 那么久执行该函数, 并且捕获控制台报错
      • 注意: 无论是成功还是失败, 回调函数都可以通过声明形参来接受promise 对象的结果值

    • . then 的返回值

      • Promise 对象, 简称 p2

      • 如何控制 p2的状态

        • 变为成功

          • 只要回调函数中的代码, 成功执行结束就会变成 p24
          • 如果回调函数中的代码成功的promise 对象, 那么 p2 变为成功

        • 变为失败

          • 只要回到函数中的代码, 出现报错, 无法执行结束, 那么 p2就会变成失败
          • 如果回调函数中返回失败的 promise对象, 那么 p2变为成功

        • 继承状态

          • 如果监视的promise 状态发生了变化, 但是没有对应的回调函数可以执行, 那么 p2 的状态就会参考 p1 的状态继续往下

        • 注意: 无论回调函数中返回的任何内容, 都不会影响 .then返回一个 promise 对象这个事情

        • 如果在回调函数中返回一个非 promise对象, 那么该数据就会成为p2 的 resolve值

        • .then方法, 不是一调用就开启微任务, 而是必须被监视的 promise变化状态, 才会开启微任务

  • .catch (它是 .then的语法糖, 相当于 .then 没有传成功的回调)

    • 用处: 捕获 promise 出现的错误
      • 监视某个 promise 状态的变化, 如果为失败状态, 就会执行回调函数
      • 数据类型: 函数
      • 接收参数个数: 1个
        • 数据类型函数 (失败的回调)

async / await

async 和 await 其实是 promise + .then的语法糖

async 和 await 是解决回调地狱的终极方案

async 和 await 实现了让开发者可以同步的流程书写异步的代码

async 关键字

  • async 可以脱离 await 单独使用
  • async 关键字需要在函数声明之前使用
  • async 的返回值一定是一个全新的 promise 对象(简称 p1)
    • 变为成功
      • 只要 async 函数中的代码成功执行那么就是 成功
      • 只要 async 函数返回一个成功的 promise 对象. 那么就是成功
    • 变为失败
      • 只要 async 函数中的代码出现报错, 无法正常执行, 就是失败
      • 如果async 函数返回一个失败的 promise 对象, 那么就是失败
      • await 右侧的 promise 是失败的状态, 就是失败
    • 注意: async 如果 return 数据, 那么该数据就会成为 p1的结果值

await 关键字

await 不能脱离 async 单独使用

  • 但是在最新的ES语法中, await 已经允许在顶层作用域中单独使用
  • await 关键字可以拦截后续代码的执行, 但是范围仅限于 async 函数内部
    • 其实 awiat 就是 .then
  • 注意:
    • await 虽然会拦截后续代码执行, 但是右侧的代码还是会同步执行的
    • 无论 await 右侧的 promise 是成功还是延迟成功, 后续代码都是在 微任务执行
  • await 表达式的结果, 就是右侧 promise 对象的结果值
    • await 表达式右侧结果如果是一个非 promise 对象, 那么就会将数据作为 await的返回值

promise.all()

批量一次性发生多个异步请求

当都成功, 返回的 promise 才成功

一旦有一个失败, 返回的 promise就失败了

返回值, 全新的 promise 对象

promise.rece()

批量一次性发送多个异步请求

谁先回来就用谁的结果

返回全新的 promise 对象, 只要一个 promise 成功, 它就会变为成功

async 和 await 与 promise 的关系

async / awiat 是消灭异步回调的终极武器, (以同步的流程, 书写异步的代码)

作用: 简化 promise 对象的使用, 不用在使用 then catch 来指定回调函数

但是和 promise 并不互斥

反而两者相辅相成

执行 async 函数, 返回 promise 对象

await 相当于 promise 的 then

try.catch 可捕获异常, 相当于 promise 的 catch

DOM事件机制

是什么

  • 事件机制

事件是什么

  • 事件就是在程序运行过程中发生的事情或者发生的操作
  • 简单来说: 用户在项目中做了什么事情

事件三要素

  • 事件源
  • 事件类型 (事件名)
  • 事件处理函数 (事件处理函数)
  • 例如 div.onclick = function() {}
  • 有多重事件组成的一套执行顺序的规则就是事件机制

事件机制分为几种

  • 事件捕获机制 (网景)
  • 事件冒泡机制 (IE)
  • 标准事件机制 (标准事件流) 现如今绝大多数浏览器都是这个机制

标准事件机制分为几个阶段

  • 捕获阶段
    • 从最外层元素 document 开始, 向内逐层触发同类型事件, 直到最内层的目标元素为止
  • 目标阶段
    • 将目标元素身上的同类型事件的事件处理全部触发
  • 冒泡阶段
    • 从最内层目标元素开始, 向外层逐层触发同类型事件, 直到最外层的 document 为止

怎么用

  • 绑定事件监听语法

    • DOM 0 写法

      • div.onclick = function () {} 解绑 div.onclick = null
      • 本质给 div 对象的onclick 赋值操作
      • 优点: 语法简单, 绑定简单, 解绑也简单
      • 缺点: 由于是将处理器函数放在一个属性里面, 所以即便绑定多次也只能保存最后一个处理器函数

    • DOM 2 写法

      • div.addEventListener(“click”, function() {})
      • 本质: 在调用 div 对象身上的 addEventListener方法
      • 优点: 由于函数可以调用多次, 可以一个事件绑定多个处理器函数
      • 缺点: 语法麻烦, 绑定麻烦, 解绑麻烦, 解绑的时候, 必须跟绑定时是同一个
      • 第三个参数
        • 数据类型: 布尔值 || 对象
        • 如果是布尔值
          • true: 将事件绑定在捕获阶段
          • false: 将事件绑定在冒泡阶段(默认)
        • 对象
          • passive属性:
            • true: 代表处理器中的 preventDefault 可以生效
            • false: 代表处理器中 preventDefault 不生效
          • capture属性:
            • true: 将事件绑定在捕获阶段
            • false: 将事件绑定在冒泡阶段
      • 如何阻止冒泡
        • event,cancelBubble = true ie专用
        • event,stopPropagation 高级浏览器专用
      • 如何阻止捕获
        • event.stopPropagation () 用在哪个阶段就是哪个阶段继续触发
      • 在哪儿用
        • vue 后台管理系统编辑模式

事件委托

事件委托是什么

  • 事件委托其实就是将本来绑定在子节点身上的事件处理器绑定到父节点身上, 这类操作就叫做事件委托
  • 事件委托其实就是开发者根据事件机制, 研究出来的一种性能优化的写法
  • 利用到了事件会逐层向外冒泡的原理

为什么要使用事件委托

  • 问题: 没使用事件委托之前会出现什么问题

    • 如果需要绑定的节点非常多, 那么就需要给每一个节点都准备一个事件处理函数
    • 后续的节点, 没有办法使用到之前绑定的事件

  • 使用事件委托之后得到的效果

    • 无论具有多个子节点, 都只需要创建一个事件回调函数, 占用内存减少
    • 后续新增的节点也可以享受到之前绑定的事件

怎么用

  • 封装一个全新的函数, 用于接受四个数据
    • 事件源
    • 事件名
    • 事件处理器
    • 指定元素的选择器字符串
  • 在事件源身上绑定事件, 并将一个全新的函数作为事件真正的处理器
  • 在事件处理器中, 我们会判断当前目标元素是否符合选择器字符串的要求
  • 如果满足选择器字符串的要求, 就会使用 call方法调用全新函数的事件处理器, 否则
